# Markdown 文档元数据与免责规范（Skill）

## Overview

本 Skill 约定：项目中**新增或实质性修订**的 Markdown（.md）文件，尤其是 `docs/` 下的 PRD、设计、说明类文档，**必须**包含：(1) **最新修订日期时间**（精确到秒，即创建或修改该文件时的时间）；(2) **免责/时效性说明**。便于读者判断文档时效、避免将历史描述误当作当前系统行为。

**与 `codex/rules/docs-md-metadata.md` 的关系**：规则文件给出强制约束与文首示例；本 Skill 提供完整模板、示例、反例与可选放置方式，便于执行与 Code Review。

## Core Principles

### 1. 最新修订日期时间

- 每个此类 .md 文档**必须**在明显位置标注该文档的**最新修订日期时间**，**精确到秒**，即该文档创建或修改时的实际时间。
- 每次对文档做实质性修订时，**必须**更新该日期时间（或增加一条修订记录）。
- 推荐位置：文档开头（标题下方）或文末「修订记录」小节。

### 2. 免责说明

- 每个此类 .md 文档**必须**包含一句**免责或时效性说明**，明确提示：系统实现或产品行为可能已变化，读者应以当前代码与产品为准，谨慎参考文档。
- 推荐位置：文首说明块内、正文之前，与修订日期一起呈现。

### 3. 适用范围

- **必须遵守**：`docs/` 下新增或大改的 PRD、设计、架构、说明类 .md（如 `docs/prd/`、`docs/engineering/`、`docs/design/` 等）。
- **建议遵守**：项目根目录或各模块下的 README、CONTRIBUTING 等正式说明文档。
- **可简化**：代码内联注释、临时笔记、仅个人使用的 .md 可仅标日期或省略，但建议至少注明「非权威/可能过期」。

## Implementation Patterns

### 文首块标准写法（推荐）

```markdown
# 文档标题

**成文日期**：2026-02-01 14:30:00 UTC+8
**最后修订**：2026-02-01 14:30:00 UTC+8

本文档说明……（目的与存放路径，可选一两句）。

**阅读时当前系统实现可能已发生变化，请以实际代码与产品行为为准，谨慎参考。**

---

## 一、正文标题
```

### 仅日期时间 + 免责（极简）

```markdown
# 标题

**最后更新**：2026-02-01 14:30:00 UTC+8。阅读时系统可能已变更，谨慎参考。

---
```

### 文末修订记录（可选，与文首日期时间二选一或并存）

```markdown
## 修订记录

| 日期时间                | 说明     |
| ----------------------- | -------- |
| 2026-02-01 14:30:00 UTC+8 | 初稿；补充 Studio 集群语义与 agent_id 说明。 |
```

## Common Anti-Patterns

### 无日期

```markdown
# 某设计说明

本文档描述……

**谨慎参考。**
```

- **问题**：未标注修订日期时间，读者无法判断文档时效。
- **改正**：在文首或文末增加「成文日期」或「最后修订」（精确到秒）并随修订更新。

### 无免责说明

```markdown
# 某 PRD

**成文日期**：2026-02-01 14:30:00 UTC+8

## 一、需求
```

- **问题**：未声明文档可能与当前系统不一致，易被当作权威现状。
- **改正**：在文首块内增加一句免责说明（见上文表述示例）。

### 免责表述含糊

```markdown
本文档可能有过时内容。
```

- **问题**：「可能有过时」未明确建议以代码/产品为准，约束力不足。
- **改正**：使用「请以实际代码与产品行为为准，谨慎参考」或等价表述。

## Best Practices

1. **成文与修订分离**：既有「成文日期」也有「最后修订」时，修订日期时间（精确到秒）随每次实质性修改更新。
2. **统一表述**：同一项目内免责说明措辞可统一（如采用 Rule 中的推荐句），减少歧义。
3. **Code Review**：在 PR 中新增或大改 .md 时，检查是否满足「日期时间（精确到秒）+ 免责」两项。
